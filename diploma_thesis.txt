CHAPTER 4. IMPLEMENTATION AND DEVELOPMENT


4.1 Technology Stack


The NeuroBalance system is implemented as a distributed, microservice-based application comprising a mobile iOS client, three independently deployable Spring Boot services, an asynchronous messaging layer, a Python-based machine learning inference service, and three isolated PostgreSQL databases. Each layer was selected based on demonstrated performance in production-grade systems and alignment with the functional requirements of a real-time cognitive health monitoring platform.


4.1.1 Backend Technologies


Spring Boot and Java 17

The backend is written in Java 17 using Spring Boot — NBAuthService (version 4.0.2, port 8081) and NBCheckinService (version 4.0.2, port 8082) handle core business logic, while NoteAI Service (version 3.5.0, port 8083) manages AI-powered note summarisation. Spring Boot is a production-ready extension of the Spring Framework that provides convention-over-configuration defaults, an embedded Tomcat server, dependency injection via the application context, and seamless integration with data access, security, and messaging libraries.

Spring Framework 6.2 introduces enhanced support for reactive programming and virtual threads under Project Loom (JEP 425), though this project uses the standard thread-per-request model for predictable latency. Java 17 is a Long Term Support (LTS) release with sealed classes, enhanced pattern matching, and improved garbage collection via the G1 GC, which reduces pause times for heap sizes typical in microservice containers (128–512 MB).

Spring Security 6 and JWT Authentication

Authentication across all services uses stateless JSON Web Tokens (JWT) conforming to RFC 7519. Tokens are signed with the HS256 (HMAC-SHA-256) algorithm using a shared secret configured via environment variables. Each token carries a subject claim containing the numeric userId and expires after 86,400 seconds (24 hours). The NBAuthService generates tokens on successful login via the jjwt library (version 0.12.6), while all other services validate incoming Authorization: Bearer headers through a JwtAuthFilter that extends OncePerRequestFilter.

Password storage follows NIST SP 800-63B guidelines: BCrypt with a work factor of 10 rounds is used for all password hashes. BCrypt's adaptive cost factor ensures that hashing time increases as hardware improves, protecting stored credentials even if a database breach occurs. The Spring Security PasswordEncoder interface abstracts the hashing mechanism, making future algorithm upgrades non-breaking.

PostgreSQL 15 and Database-per-Service Pattern

The system employs three logically separate PostgreSQL 15 databases: nb_auth (port 5433), nb_checkin (port 5435), and nb_noteai (port 5436). This Database-per-Service pattern, described by Newman (2015) as foundational to microservice design, eliminates shared schema coupling and enables each service to evolve its data model independently. Cross-service data references (e.g., userId in nb_checkin tables) are soft references with no enforced foreign-key constraints across databases, accepting eventual consistency in exchange for decoupled deployability.

Spring Data JPA with Hibernate 6 provides the ORM layer. HikariCP serves as the JDBC connection pool, configured with a maximum pool size of 10 and a connection timeout of 30 seconds. Database schema migrations are managed by Flyway, ensuring reproducible schema state across environments through versioned migration scripts (V1__, V2__, etc.).

Apache Kafka — Asynchronous Event Streaming

Health metrics computation is fully decoupled from the check-in submission request via Apache Kafka (Confluent Platform 7.6). When a user submits a daily check-in, NBCheckinService publishes a CheckInEvent to the topic checkin.created (3 partitions, replication factor 1) using KafkaProducerService. The response to the iOS client is returned immediately with HTTP 201, without waiting for ML inference.

A HealthMetricsKafkaConsumer (also within NBCheckinService) subscribes to the same topic, retrieves the full check-in record from the database, invokes the ML Service via HTTP, and persists the resulting health metrics. This pattern ensures that the user-facing API response latency is bounded at the database write time (~15–30 ms), while ML inference (which may take up to 200 ms) occurs asynchronously.

Kafka's partition-based ordering guarantees that events for a given userId are processed sequentially within a partition (when keyed by userId), preventing race conditions in metric computation.

Groq Cloud API — Large Language Model Integration

NoteAI Service integrates with the Groq Cloud API to generate concise AI summaries of user journal notes. Groq provides inference for Meta's Llama 3.x family of models with sub-second latency due to LPU (Language Processing Unit) hardware acceleration. The integration uses Spring AI (spring-ai-openai-spring-boot-starter, version 1.0.0-M6), which abstracts the HTTP client and prompt templating. Requests are made with a temperature of 0.3 to produce deterministic, factual summaries rather than creative outputs.

Docker and DigitalOcean Deployment

All services are containerised using Docker and orchestrated with Docker Compose, totalling 10 containers: three Spring Boot services, three PostgreSQL instances, one ML service (Flask), Apache Kafka, Zookeeper, and a Swagger documentation container. The stack is deployed on a DigitalOcean Droplet. Environment-specific configuration (database URLs, JWT secrets, Kafka broker addresses) is externalised via Docker Compose environment blocks, following the twelve-factor app methodology.

API documentation is generated automatically by springdoc-openapi 2.8.8 and exposed at /swagger-ui.html on each service. Swagger UI enables integration testing directly from the browser without requiring a separate HTTP client.


4.1.2 iOS Frontend Technologies


Swift 5.9 and SwiftUI

The NeuroBalance iOS application targets iOS 16+ and is written in Swift 5.9 using the SwiftUI declarative UI framework. SwiftUI's state-driven rendering model eliminates the need for imperative view updates; UI components are functions of application state, and any state change automatically triggers a re-render of the affected views. This reduces the surface area for UI inconsistency bugs that are common in UIKit-based imperative layouts.

The project adopts the MVVM (Model-View-ViewModel) architectural pattern. Views observe @Published properties on ViewModels via @StateObject or @ObservedObject property wrappers. ViewModels encapsulate all business logic, network calls, and state transformations, while Views remain pure rendering functions. This separation enables unit testing of ViewModels without launching a UI.

Async/Await Networking

All network communication uses Swift's structured concurrency model introduced in Swift 5.5 (SE-0296). URLSession's async/await variants (data(from:) and data(for:)) replace callback-based networking, producing linear, readable code without callback pyramids. API errors are propagated via Swift's typed throws mechanism, allowing ViewModels to differentiate between network errors, authentication failures (HTTP 401), and server errors (HTTP 5xx) with exhaustive switch statements.

JWT tokens received from NBAuthService are stored in the iOS Keychain via the Security framework, using kSecClassGenericPassword items. Keychain storage encrypts tokens at rest using the device's Secure Enclave and is not accessible to other apps, satisfying OWASP MASVS-STORAGE-1 requirements.

Combine and Reactive State Management

Where real-time or debounced input handling is needed, the Combine framework provides reactive operators (debounce, map, filter). Form validation in the registration and login screens uses Combine pipelines to validate email format and password strength reactively as the user types, providing immediate visual feedback without blocking the main thread.

Offline-First Design

The application maintains a local cache of the most recently fetched health metrics and check-in status using UserDefaults for lightweight scalar values and Core Data for structured records. When the device is offline, the UI renders cached data with a visual indicator. On reconnection, a background URLSession task synchronises pending check-ins to the server.


4.1.3 Machine Learning Technologies


Overview and Motivation

The NeuroBalance ML subsystem predicts cognitive wellness scores and generates personalised behavioural recommendations from daily self-reported check-in data. Two gradient-boosted tree models — XGBoost for regression and CatBoost for classification — were selected based on their superior performance on tabular data demonstrated in published benchmarks (Shwartz-Ziv & Armon, 2022).

The relationship between lifestyle factors and cognitive performance is well-established in sleep science literature. Killgore (2010) demonstrated that sleep deprivation of even one night produces significant deficits in executive function, attention, and emotional regulation, with effects comparable to acute alcohol intoxication at 0.05% BAC. Walker (2017) further established that REM sleep is specifically linked to emotional memory consolidation, while deep (slow-wave) sleep drives declarative memory consolidation. These findings justify using sleep quality and duration as primary features in the cognitive scoring model.

Feature Engineering

The ML Service receives 15 raw input variables from NBCheckinService:

    sleepHours, sleepQuality, deepSleepMinutes, remSleepMinutes, bedtime, wakeTime, energyLevel, morningMood, eveningMood, stressLevel, didExercise, ateHealthy, hadSocialInteraction, screenTime, hydrationGlasses

The function calc_engineered_features() transforms these into 23 engineered features. Key transformations include:

    sleep_efficiency = total_sleep_minutes / time_in_bed_minutes
    sleep_debt = max(0, 480 - total_sleep_minutes)
    mood_swing = abs(eveningMood - morningMood)
    recovery_index = (sleepQuality * 0.4 + energy_level * 0.3 + (10 - stressLevel) * 0.3)
    lifestyle_score = (didExercise * 33) + (ateHealthy * 33) + (hadSocialInteraction * 34)

These composite features capture interactions between variables that raw inputs cannot express individually. For example, sleep_efficiency distinguishes between a person who sleeps 7 hours in 7 hours (high efficiency) and one who sleeps 7 hours in 10 hours (fragmented, low-quality sleep).

Engineered features are normalised using a pre-fitted StandardScaler (scikit-learn), applying z-score standardisation (x - mean) / std computed on the training dataset. This is critical for gradient-boosted trees when features have very different scales (e.g., sleep_hours in range 0–12 vs. lifestyle_score in range 0–100).

XGBoost Cognitive Score Model

Chen and Guestrin (2016) introduced XGBoost (Extreme Gradient Boosting) as a scalable, regularised implementation of gradient boosting that outperformed all other methods on 29 of 32 Kaggle challenge winning solutions at the time of publication. XGBoost builds an ensemble of shallow decision trees additively, with each tree trained to correct the residual errors of the previous ensemble. The objective function combines a loss term (MSE for regression) with L1 and L2 regularisation to prevent overfitting.

In NeuroBalance, the XGBoost model predicts a continuous cognitive_score in the range 0–100. The model was trained on synthetic lifestyle-wellness data generated using physiologically validated distributions derived from sleep science literature. Training used 5-fold cross-validation with early stopping (50 rounds) to select the optimal number of boosting rounds. The final model achieves an RMSE of 6.8 on held-out validation data.

XGBoost inference on 23 features takes approximately 2–5 ms on the Flask server, well within the asynchronous processing budget.

CatBoost Recommendation Classifier

Prokhorenkova et al. (2018) introduced CatBoost (Categorical Boosting) as a gradient boosting variant that handles categorical features natively through ordered target statistics, eliminating the need for one-hot encoding and reducing target leakage. CatBoost demonstrated state-of-the-art performance on heterogeneous tabular datasets with mixed categorical and numerical features.

In NeuroBalance, the CatBoost model classifies each user's current state into one of four recommendation categories:
    SLEEP_INCREASE    — when sleep < 6 hours or quality < 5
    STRESS_DECREASE   — when stress > 7 and mood < 5
    EXERCISE_INCREASE — when didExercise = false for 3+ consecutive days
    SCREEN_DECREASE   — when screenTime > 6 hours

The SmartRecommendationSelector class in the ML service applies rule-based post-processing on top of CatBoost's probability output, using a minimum confidence threshold of 0.45 to suppress low-confidence recommendations. The /recommend/top3 endpoint returns ranked recommendations; /recommend/single returns the highest-confidence recommendation for UI display.

Health Metrics Computation

Three scalar health metrics are computed within NBCheckinService (not by the ML model) using deterministic formulae:

    M-Rest (0–100): Reflects sleep quality and duration.
        mRest = (sleepHours / 8.0) * 40 + (sleepQuality / 10.0) * 40 + (deepSleepMinutes / 90.0) * 20
        Clamped to [0, 100].

    M-Ready (0–100): Reflects readiness for cognitive tasks.
        mReady = (energyLevel / 10.0) * 40 + (morningMood / 10.0) * 35 + (mRest / 100.0) * 25

    M-Balance (0–100): Reflects emotional and lifestyle balance.
        mBalance = ((10 - stressLevel) / 10.0) * 40 + lifestyleScore * 0.35 + (morningMood / 10.0) * 25

    overallWellnessScore = round((mRest + mReady + mBalance) / 3.0)

These formulae are grounded in the research of Stults-Kolehmainen and Sinha (2014), who showed that exercise, social interaction, and sleep quality together account for the majority of variance in self-reported psychological wellness.

Service Architecture

The ML Service is implemented in Python 3.11 using the Flask 3.0 microframework, with Flasgger providing Swagger documentation at /apidocs. The service is accessible only via Docker's internal network (not exposed on the host), and NBCheckinService calls it via HTTP POST /predict after consuming a Kafka event. This internal-only exposure prevents direct external access to the ML inference endpoint.

Model files (xgboost_model.pkl, catboost_model.pkl, scaler.pkl) are loaded once at Flask startup and held in memory, ensuring that subsequent inference calls do not incur disk I/O. The /predict endpoint processes one record at a time; batch processing endpoints are not required for the current usage pattern.


4.2 Implementation


4.2.1 Backend Implementation


User Registration and Authentication

The registration flow in NBAuthService begins with input validation using Jakarta Bean Validation annotations (@Email, @NotBlank, @Size). If validation passes, BCryptPasswordEncoder hashes the plaintext password, and the user record is persisted via UserRepository (a Spring Data JPA repository extending JpaRepository<User, Long>). The response returns the created userId without exposing the password hash.

The login flow retrieves the user by email, verifies the password with BCrypt, and constructs a JWT using JwtUtil:

    String token = Jwts.builder()
        .subject(String.valueOf(user.getId()))
        .issuedAt(new Date())
        .expiration(new Date(System.currentTimeMillis() + 86_400_000))
        .signWith(secretKey, Jwts.SIG.HS256)
        .compact();

The JwtAuthFilter extracts the token from the Authorization header, validates signature and expiry, and sets a UsernamePasswordAuthenticationToken in the SecurityContextHolder, making the userId available to all downstream controller methods via @AuthenticationPrincipal.

Daily Check-in and Kafka Publishing

The DailyCheckInService.createCheckIn() method performs three operations within a single @Transactional boundary: it saves the DailyCheckIn entity, updates the UserStreak record (incrementing current_streak if last_checkin was yesterday, or resetting to 1 otherwise), and awards XP to the USER_CHARACTER. After the transaction commits, KafkaProducerService publishes a CheckInEvent:

    kafkaTemplate.send("checkin.created", userId.toString(),
        new CheckInEvent(userId, checkInDate));

The Kafka message key is the userId string, ensuring all events for the same user route to the same partition for ordered processing. The KafkaTemplate is configured with StringSerializer for keys and JsonSerializer for values.

Health Metrics Kafka Consumer

HealthMetricsKafkaConsumer is annotated with @KafkaListener(topics = "checkin.created", groupId = "health-metrics-group"). On receiving a CheckInEvent, it:

    1. Queries DailyCheckInRepository and SleepLogRepository for the given userId and date.
    2. Computes mRest, mReady, mBalance using the formulae in Section 4.1.3.
    3. Sends an HTTP POST request to the ML Service's /predict endpoint with 15 input features.
    4. Receives the cognitiveScore and recommendations in the response JSON.
    5. Persists a HealthMetrics entity with all four scores.

If the ML Service is unavailable (connection timeout or HTTP 5xx), the consumer falls back to saving metrics computed from the deterministic formulae only, with a null ML score. This graceful degradation ensures that users still see wellness metrics even during ML service restarts.

Gamification System

The XP and level progression system uses an exponential curve to prevent early-game level inflation. The XP threshold to reach level N is defined as:

    XP_threshold(N) = N^2 * 100

This means Level 2 requires 400 XP, Level 3 requires 900 XP, Level 4 requires 1,600 XP, and so on. XP is earned through:

-------------------------------------------------------------------------------------
Activity                        XP Earned       Frequency Cap
-------------------------------------------------------------------------------------
Daily check-in submission       50 XP           Once per day
Streak milestone (7 days)       200 XP          Once per milestone
Streak milestone (30 days)      500 XP          Once per milestone
Brain game completion           10–40 XP        Based on difficulty and score
Perfect score on brain game     +25 XP bonus    Per game
-------------------------------------------------------------------------------------

Brain game difficulty is adaptive: if the cognitive score (from XGBoost) is above 75, HARD difficulty is selected; scores 50–75 select MEDIUM; scores below 50 select EASY. This ensures that cognitive challenges scale with the user's current mental state, implementing the flow theory principle of matching challenge to skill level (Csikszentmihalyi, 1990).


4.2.2 iOS Frontend Implementation


Dashboard and Health Metrics Display

The HealthMetricsViewModel fetches today's health metrics from GET /health-metrics/today after the user completes a check-in. The response is decoded into a HealthMetricsResponse Decodable struct:

    struct HealthMetricsResponse: Decodable {
        let mRest: Int
        let mReady: Int
        let mBalance: Int
        let overallWellnessScore: Int
        let recommendations: [String]
    }

The dashboard renders three circular progress rings (SwiftUI's Circle with trim modifier) for M-Rest, M-Ready, and M-Balance, colour-coded by score range: green (80–100), amber (50–79), red (0–49). The overall wellness score is displayed as a large centred numeral with a label indicating the cognitive state (Excellent / Good / Fair / Low) derived from XGBoost's output via the recommendations field.

Check-in Flow

The check-in form is a multi-step SwiftUI view using a @State variable for the current step index. Each step collects a subset of inputs (mood sliders, sleep duration picker, habit toggles). On the final step, the ViewModel calls POST /checkins and, on success, triggers a local notification reminding the user to check in the following morning at 8:00 AM via UNUserNotificationCenter.

Character and Progression Screen

The character progression screen reads from GET /streaks/me and GET /auth/me to display the user's current level, XP progress bar, and streak count. The SwiftUI animation system (withAnimation(.spring())) provides a smooth XP bar fill on level-up events, with a haptic feedback pulse via UIImpactFeedbackGenerator.


4.3 Prototype Demonstration


The NeuroBalance prototype was iteratively tested on iOS 17 devices running the TestFlight build. The following screens constitute the complete user journey:

Screen 1 — Onboarding and Registration
The first screen presents the NeuroBalance brand identity. The user enters their email address and a password (minimum 8 characters, validated in real-time by a Combine pipeline). On submission, the app calls POST /auth/register, stores the returned JWT in Keychain, and navigates to the onboarding questionnaire.

Screen 2 — Onboarding Questionnaire
A multi-page questionnaire collects: age, gender, primary wellness goals (multiple selection: sleep, focus, stress, fitness), and current concerns. This data is submitted to POST /auth/onboarding and used to configure initial recommendation weights in the ML service.

Screen 3 — Home Dashboard
The main dashboard displays the date, current streak, and a three-ring wellness summary. Each ring shows one of M-Rest, M-Ready, or M-Balance as a percentage. Below the rings, the top recommendation from CatBoost is shown as a card with a short explanatory text and an action button.

Screen 4 — Daily Check-in (Step 1: Mood)
Two sliders collect morning mood and current energy level (1–10 scale). The slider handles use a custom SwiftUI ViewModifier for a smooth gradient thumb appearance consistent with the brand palette.

Screen 5 — Daily Check-in (Step 2: Sleep)
A time picker collects bedtime and wake time. The calculated sleep duration is displayed dynamically as the user adjusts the pickers. A quality slider (1–10) and a boolean toggle for felt rested complete this step.

Screen 6 — Daily Check-in (Step 3: Habits)
Three toggle buttons (Exercise, Healthy Eating, Social Interaction) with icon illustrations are presented. Stress level is collected via a colour-gradient slider that transitions from green (low stress) to red (high stress).

Screen 7 — Check-in Confirmation
A celebratory animation (Lottie) plays on successful check-in submission. The XP earned and streak update are displayed with a spring animation. A counter shows time until tomorrow's check-in window opens.

Screen 8 — Health Metrics Detail
This screen appears once the asynchronous health metrics calculation completes (typically within 2–3 seconds of check-in submission). Three metric cards display M-Rest, M-Ready, and M-Balance with trend arrows comparing to the previous day's scores.

Screen 9 — Recommendations
The full ranked list of CatBoost recommendations is presented as expandable cards. Each card explains the recommendation (e.g., "Your sleep duration was below your target. Consider moving your bedtime 30 minutes earlier") with a link to a tip article.

Screen 10 — Brain Games Hub
A grid of available brain games (memory match, reaction time, pattern recognition) is displayed. Each game shows the user's personal best score and the current difficulty level (EASY/MEDIUM/HARD) determined by their latest cognitive score.

Screen 11 — Notes (NoteAI)
A list of the user's journal entries is shown. Each note card displays the AI-generated summary (from Groq Llama 3.x) below the note title, truncated to two lines. A compose button opens the note editor.

Screen 12 — Note Editor with AI Summary
The note editor provides a plain-text input area. On save, the app submits the note to POST /notes. The server calls Groq's API to generate a 2–3 sentence summary, which appears below the note body within approximately 1–2 seconds. Tags extracted by the AI are shown as chips below the summary.


CHAPTER 5. BUSINESS MODEL


5.1 Overview

NeuroBalance operates as a B2C (Business-to-Consumer) digital health application targeting working professionals, university students, and individuals seeking to quantify and improve their cognitive wellness. The product addresses a documented gap in the market: while fitness wearables measure physical activity, no mainstream mobile application provides a continuous, ML-driven cognitive wellness index derived from daily self-reported lifestyle data.

The global mental wellness app market was valued at USD 5.2 billion in 2023 and is projected to grow at a CAGR of 16.5% through 2030, driven by increased awareness of mental health, widespread smartphone adoption, and post-pandemic attention to psychological resilience (Grand View Research, 2024). NeuroBalance is positioned within the cognitive performance sub-segment, competing with general wellness apps (Headspace, Calm) while offering a quantitative, personalised scoring layer that competitors lack.


5.2 Value Proposition

For individual users:
    A daily cognitive wellness score (0–100) derived from sleep, mood, stress, and lifestyle data.
    Personalised ML-generated recommendations grounded in published sleep science research.
    Gamification mechanics (XP, streaks, brain games) that drive daily engagement and habit formation.
    AI-powered journal with automatic note summarisation via Groq's Llama 3.x.

For enterprise clients (B2B tier, Phase 2):
    Aggregated, anonymised team wellness dashboards for HR departments.
    Integration with calendar and productivity APIs to correlate cognitive scores with work output.
    EAP (Employee Assistance Programme) feature bundles for corporate wellness programmes.


5.3 Target Segments

Primary segment: Knowledge workers aged 22–40 in Kazakhstan and Central Asia experiencing stress, sleep issues, or declining focus. This segment is characterised by high smartphone usage (4+ hours/day), willingness to track health data, and disposable income for subscription apps.

Secondary segment: University students, particularly in STEM programmes, who use the app during exam periods to optimise sleep and stress management.

Tertiary segment (future): Corporate HR departments procuring wellness software for teams of 50+ employees.


5.4 Revenue Model

Freemium with subscription:
    Free tier: Check-in tracking, basic mood history, 3 brain games, 5 notes/month.
    NeuroBalance Premium (USD 4.99/month or USD 39.99/year): Full ML metrics, all brain games, unlimited notes with AI summaries, streak rewards, character progression, 30-day history analytics.

B2B enterprise licences (Phase 2):
    Per-seat pricing at USD 3–5/employee/month with a minimum of 50 seats, including API access and HR dashboard.

In-app character cosmetics (non-pay-to-win):
    Cosmetic-only virtual items for character customisation, priced at USD 0.99–2.99. These do not affect XP or game scores, maintaining competitive fairness.


5.5 Go-to-Market Strategy

Phase 1 (Months 1–6): Beta launch via TestFlight in Kazakhstan. Target: KBTU students and IT community via university channels, LinkedIn, and Telegram wellness communities. Goal: 500 active users, validate retention metrics (D7 and D30 retention targets: 40% and 20% respectively).

Phase 2 (Months 7–18): App Store launch in Kazakhstan and Russia. Localisation for Russian language. Onboard first 3 corporate clients. Goal: 5,000 monthly active users, 8% premium conversion rate.

Phase 3 (Months 19–36): Expand to CIS countries. Integrate HealthKit (Apple Health) for automatic sleep and activity data import, reducing manual input friction. Explore partnerships with health insurance providers for premium subsidisation.


5.6 Key Performance Indicators

    Monthly Active Users (MAU): Target 5,000 by Month 18.
    Daily Active Users / MAU ratio (DAU/MAU, "stickiness"): Target 35% (industry benchmark for health apps: 30%).
    D7 Retention: Target 40%. D30 Retention: Target 20%.
    Average Revenue Per User (ARPU): Target USD 2.50/month blended (free + paid).
    Premium Conversion Rate: Target 8% of free users converting within 90 days.
    Net Promoter Score (NPS): Target 45+ (above industry average of 32 for health apps).
    Mean check-in completion rate: Target 70% of registered users completing at least one check-in per week.


5.7 Cost Structure

    Cloud infrastructure (DigitalOcean): USD 48–96/month for Droplet and managed database tiers.
    Groq API: USD 0.00027/1,000 tokens (Llama 3.8B input). At 1,000 notes/day, estimated USD 8/month.
    App Store and Google Play developer fees: USD 99/year (Apple), USD 25 one-time (Google).
    Engineering (solo or small team): Primary cost driver in Phase 1.
    Marketing: USD 200–500/month for targeted Instagram and Telegram ads in Phase 1.

Total estimated monthly burn (Phase 1): USD 350–700 (infrastructure + marketing; engineering is founder-hours).

5.9 Risk Analysis

Technical risk: The ML models are trained on synthetic data. As real user data accumulates, models should be retrained to reflect actual user populations. Mitigation: Implement online learning pipeline in Phase 2, with user consent for anonymised model training.

Regulatory risk: Health data processing in Kazakhstan falls under the Law on Personal Data and its Protection (2013). Mitigation: All data stored on DigitalOcean servers with optional in-country hosting; privacy policy drafted with explicit consent flows.

Retention risk: Health apps typically experience significant churn after 30 days. Mitigation: Gamification loop (streaks, XP, character progression) is specifically designed to create habit triggers (Fogg, 2009; Eyal, 2014); push notifications are personalised based on cognitive score trends.

Scalability risk: Current single-droplet deployment has a practical limit of ~1,000 concurrent users. Mitigation: Docker Compose configuration can be migrated to Kubernetes with horizontal pod autoscaling; PostgreSQL can be upgraded to managed DigitalOcean database clusters with read replicas.
